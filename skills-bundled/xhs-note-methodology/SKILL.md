# xhs-note-methodology

> 这是一个 **参考型 skill**：它不执行命令，只提供一套被实战验证过的小红书内容方法论，让 Agent 在做
> **博主分配 / 方案策划 / 审稿 / 数据解读** 时有稳定的思考框架。

Agent 在任何 **ALLOCATION / BLOGGER_BRIEF / CONTENT_REVIEW** 会话里遇到以下情况应该先回看本 skill：

- 用户问 "这个博主适不适合投放？" → 用第 2 节「博主 7 维评估表」
- 用户说 "帮我写一个种草视频脚本" → 用第 4 节「爆款结构 S-H-C-R-C」
- 用户说 "审一下这个稿子" → 用第 6 节「审稿 3×3 校验清单」
- 用户要分析数据 → 用第 7 节「数据异常解读表」

---

## 1. 小红书内容生态的基本假设

1. **平台是"女性决策前搜索"而非"男性信息消费"**：用户在真正买 / 去 / 约之前来小红书交叉验证；所以内容不追时效、追"可复用"。
2. **流量公式 ≈ 标签匹配度 × 互动率 × 停留时长 × 完播/读完率**。其中互动里 **收藏 > 评论 > 点赞**（反向说明"有用"）。
3. **爆款是 80% 选题 + 15% 开头 3 秒 + 5% 内容质量**。选题错，再好看也做不起来。
4. **博主的"人设稳定度"比单篇爆款更重要**；做投放要找「稳态在 A 赛道」的博主，而不是「偶尔出过一个 A 赛道爆款」的博主。

---

## 2. 博主 7 维评估表（给 ALLOCATION 会话用）

| 维度 | 权重 | 说明 | 数据来源（DB 字段） |
|---|---|---|---|
| 赛道匹配 | 25% | 博主稳态赛道 vs 本项目投放赛道是否对齐 | `creators.track_tags_json`, `creator_accounts.category_main/sub` |
| 人设匹配 | 20% | 人设调性（「理性测评」「好朋友种草」「生活方式」等）是否匹配品牌语气 | `creators.persona_tags_json` |
| 互动率 | 15% | 近 20 条平均互动率；同粉丝段位对比而非绝对值 | `creator_accounts.engagement_rate`, `creator_posts.*` |
| 爆款率 | 15% | 近 20 条里被平台打爆款标的比例（hit_ratio）；衡量内容稳定性 | `creator_accounts.hit_ratio`, `creator_posts.is_hit` |
| 粉丝真实度 | 10% | 粉丝/互动比、涨粉曲线是否异常；用 snapshot 判断 | `creator_snapshots.*` |
| 性价比 | 10% | 报价 ÷ 历史中位互动；同档博主里横向比 | `creators.price_note`, `project_creators.quoted_price` |
| 合作履历 | 5% | 是否已合作过，上次 ROI；避免重复种草疲劳 | `project_creators` 历史记录 |

**操作指引**：
- 给每个维度 0-5 分，加权后 ≥ 3.5 = 推荐入围；2.5–3.5 = 需谈；< 2.5 = 不匹配。
- Agent 的产出必须 **按博主列出 7 维评分 + 加权总分 + 三档结论**，不要只写"推荐 / 不推荐"。

## 3. 爆款选题 4 条通则

1. **可搜索性**：标题关键词必须命中 1 条常见搜索（如「油皮夏天防晒推荐」「新手厨房装备」），Agent 可用 xhs-search-notes skill 验证关键词近 30 天热度。
2. **强对比 / 冲突**：「我花 300 试了 10 款，只有这一款…」「千万别买 XXX，这才是…」比平铺直叙的科普点击率高 2-3 倍。
3. **可复用场景**：用户能照着做、照着买、照着去的选题留存长尾更久；单纯的"日记型"爆款难复现。
4. **反常识**：把行业"默认对的"推翻一下再给出替代，最容易吃到平台的推荐流量。

## 4. 爆款视频 / 图文结构 S-H-C-R-C

> 每次让 Agent 写脚本，让它严格按这 5 段输出，拒绝"自由发挥"。

1. **S (Scene) 场景勾画**（0-3 秒）：一句话把观众拉进场景。
   > 例："油皮姐妹夏天这么干一次，底妆贴一天！"
2. **H (Hook) 钩子 / 冲突**（3-8 秒）：抛出反常识 / 对比 / 利益点。
   > 例："我试了 10 款号称油皮亲妈的粉底液，只有 2 款撑过了中午。"
3. **C (Context) 可信度**（8-20 秒）：讲"我凭什么说话"——身份、流程、横评表。**这一段直接决定收藏**。
4. **R (Recommendation) 结论 / 推荐**（20-40 秒）：清单式，**每条要带适用人群**（"敏感油皮" / "厚皮干油皮"）。
5. **C (Call-to-Action) 行动指令**（最后 5 秒）：评论区互动钩子，必须具体。
   > 例："评论区告诉我你皮肤状态 + 预算，我下期做定制推荐。"

## 5. 标题钩子库（3 档 12 种）

**节约型（适合工具类、省钱类）**
- 「这才是__的正确用法」
- 「花了 N 元才换来的__经验」
- 「99% 的人都做错了：__」

**焦虑型（适合美妆、母婴、护肤）**
- 「__姐妹别再买 __ 了」
- 「再不知道这个就晚了：__」
- 「__3 个雷区，踩了就废」

**情绪型（适合生活方式、家居、旅行）**
- 「我是怎么靠 __ 活过__的」
- 「__1 年后，我的生活…」
- 「25 岁开始__，真的来得及吗？」

**反常识型（最容易爆，但要有数据支撑）**
- 「__根本不用花钱/不用买__」
- 「贵的 __ 真的不一定好」
- 「__ 的 3 个真相，商家不会告诉你」

> **Agent 规则**：给出标题时必须同时给出钩子归档（如 `[焦虑型]`）；让 PM 一眼看出语气。

## 6. 审稿 3×3 校验清单（CONTENT_REVIEW 会话用）

### 必须改（任一条不过就打回）

1. **合规性**：夸大疗效 / 绝对化用词 / 医药器械违禁词；食品药品化妆品需走对应合规词库。
2. **事实性**：价格、成分、功效描述与产品官方资料不一致。
3. **与人设偏离**：博主人设是"理性测评"，稿子却写成"纯情绪种草"→ 必改；会降维打击博主未来流量。

### 建议改（改了更好，但不强制）

4. **开头 3 秒**：是否符合第 4 节 S-H 模式？
5. **是否有 Context 段**：直接跳结论会伤收藏率。
6. **CTA 是否具体**：泛泛的"留言讨论"改成带 2 个具体问题。

### 加分项

7. **是否埋了 2 条搜索词**：用 xhs-search-notes 验证过热度。
8. **横评表 / 清单结构**：结构化内容长尾更久。
9. **评论区预置话术**：品牌方应提前准备 3-5 条评论引导。

**Agent 输出模板**：必须用这个清单结构化输出「必须改 X 条 / 建议改 Y 条 / 加分 Z 条」。

## 7. 数据异常解读表（DATA_TRACK 会话用）

| 现象 | 最可能原因 | 下一步建议 |
|---|---|---|
| 点赞高、收藏低 | 内容娱乐性高但「没用」 | 下次多做横评 / 清单型 |
| 收藏高、评论低 | 选题对了但 CTA 不够具体 | 改 CTA 设计，带问题引导 |
| 完播率低但互动高 | 开头 3 秒没做好，但内容有干货 | 重做开头 |
| 互动率高但涨粉慢 | 内容价值高但"换人设"跳赛道 | 固定赛道 2-3 个月 |
| 同类型第 2 条远弱于第 1 条 | 平台限流 / 选题疲劳 | 换选题角度再试 |
| 粉丝数连续 7 天净下降 | 近期内容跑偏 / 被限 | 暂停投放，回看近 3 条数据 |

## 8. 人设标签体系（Creator.persona_tags 的取值约定）

为了让 Agent 和 UI 能稳定聚类，人设标签必须从下表里选（可多选）；`creators.persona_tags_json` 存一个 JSON 数组。

**语气类**：理性测评 / 好朋友种草 / 专业权威 / 搞笑段子 / 犀利吐槽 / 温柔治愈

**身份类**：学生党 / 宝妈 / 职场白领 / 自由职业 / 博主本博 / 从业者（皮肤科 / 厨师 / 律师…）

**赛道类**（可与 track_tags 重复，但 persona 侧重"博主本人被贴上的" ）：美妆测评 / 护肤科普 / 穿搭 / 家居 / 母婴 / 宠物 / 美食教程 / 探店 / 旅行 / 摄影 / 读书 / 学习方法论 / 健身 / 心理

**强度类**：高频输出 / 精品慢产 / 爆款型 / 长尾型

---

## 9. 赛道标签体系（Creator.track_tags & Post 分类）

一级赛道（固定 8 个）：美妆护肤 / 服饰穿搭 / 母婴育儿 / 美食 / 家居家装 / 出行旅行 / 教育学习 / 生活方式

> **规则**：二级赛道可自由扩展，但 Agent 写入 DB 时要保证一级赛道必在这 8 个里；避免后期做筛选时类目爆炸。

## 10. Agent 使用本 skill 的协议

Agent 在以下情况应在 system prompt 里主动引用本 skill 的小节：

```
# 使用方法论
- 挑博主：参考 xhs-note-methodology §2 7 维评估表
- 写脚本：严格按 xhs-note-methodology §4 S-H-C-R-C 结构
- 审稿：用 xhs-note-methodology §6 3×3 校验清单
- 解读数据：对照 xhs-note-methodology §7 异常表
```

引用时要 **给出具体小节编号**，让 PM 能追溯；不要拍脑袋编新框架。
